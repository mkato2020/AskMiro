## §1 — The three-layer model

Every event in the outreach pipeline lives at exactly one of three abstraction layers. Conflating them is the root cause of the queue's current contamination.

### Layer 1 — Transport

**What it represents:** SMTP-level activity. Bytes moved through mail servers.

**What it can tell you:**
- A message was transmitted
- The transmitter's address
- The recipient address(es)
- The routing metadata (Message-ID, In-Reply-To, References)
- Delivery success or failure (DSN)
- Thread continuity (same Message-ID chain)

**What it cannot tell you:**
- Whether the message was read
- Whether a human composed it
- What the sender intended

**Authoritative source:** Gmail (and the recipient's mail server)
**Evidence type:** Header inspection, deterministic
**Should the classifier write business-level state from this?** No.

### Layer 2 — Communication

**What it represents:** A human (or system acting on their behalf) composed and sent a message containing meaningful linguistic content.

**What it can tell you:**
- The sender used natural language
- The message has subject + body content
- The content can be classified as out-of-office / auto-acknowledgement / human-written
- A rough intent category if human-written (positive / negative / question / not-interested)

**What it cannot tell you:**
- Whether the sender has commercial authority
- Whether the sender's intent will translate to action
- Whether the message represents a business commitment

**Authoritative source:** Message body + headers + (probabilistic) AI inference
**Evidence type:** Mixed — deterministic for OOO/auto-reply detection, probabilistic for intent
**Should the classifier write business-level state from this?** Only with confidence stamps and audit metadata. Never directly.

### Layer 3 — Business

**What it represents:** A decision-maker at the recipient organization has expressed intent that materially affects commercial outcomes (interest, rejection, quote request, contract step).

**What it can tell you:**
- The lead is engaged, qualified, lost, or won
- The pipeline state should advance
- A revenue probability has shifted

**What it cannot tell you on its own:**
- Anything mechanical (that's transport)
- Anything linguistic (that's communication)

**Authoritative source:** Human judgment, or a clearly-versioned rules engine with operator override
**Evidence type:** Deliberate, context-aware, irreversible without audit
**Should the classifier write this directly?** Never. The classifier is allowed to *propose* state transitions. Only an authorized writer (human or explicit business-rules layer) commits them.

## §2 — What "reply" means operationally for AskMiro

The system currently has no formal definition. The deployed code treats "reply" as "any message arrived in this thread whose `From:` doesn't contain `askmiro.com`". That is a transport-layer signal masquerading as a business-layer fact.

A correct operational definition has three nested layers:

### Transport-layer reply
**A message exists in the thread whose Message-ID was not generated by AskMiro's send path.**
- Identifiable deterministically by sender domain match against our known senders (`mkato.ug@gmail.com`, `@askmiro.com`, future aliases)
- Required precondition for the next layers
- Not itself sufficient evidence of business engagement

### Communication-layer reply
**The non-AskMiro message contains human-authored linguistic content (not a bounce DSN, not an auto-reply, not a vacation responder).**
- Deterministic for the negative case (DSN headers, `Auto-Submitted` header, OOO regex patterns are reliable)
- Probabilistic for the positive case (the AI estimates "this looks like a human wrote it")
- Necessary but not sufficient for business engagement

### Business-layer reply
**The human-authored message expresses operational intent toward AskMiro: interest, rejection, question, redirection, or commitment.**
- Probabilistic — even genuine humans send ambiguous messages
- Requires confidence scoring + (eventually) operator review for non-routine intents
- The only layer at which `outreachStatus` should transition to REPLIED / QUALIFIED / NOT_INTERESTED / UNSUBSCRIBED

**The current system conflates all three.** Specifically, the deployed reply scan computes the transport-layer signal, runs an AI over the message body, and writes business-layer state — skipping the communication-layer validation entirely (DSN detection, OOO detection, self-send detection are all absent or broken).

This is why AskMiro's own follow-up emails end up in REPLIED state. The send was made from `mkato.ug@gmail.com`. The legacy filter only excludes `askmiro.com`. So Mike's own follow-up is classified as a transport-layer reply. The AI summarizes it accurately (because it accurately describes the AskMiro follow-up content). The system writes business-layer REPLIED. The operator sees "follow-up from sender" in the summary and is rightly confused.

## §3 — The nine terminal states, mapped

The states you proposed are not theoretical. They emerge directly from the empirical row content in the live queue. Each maps to a specific layer's failure mode, has a specific authorized writer, and has a specific evidence type.

| State | Layer | Failure mode it represents | Authorized writer | Evidence type | Auto-detectable? |
|---|---|---|---|---|---|
| `SELF_FOLLOWUP` | Transport | Our own send classified as a reply because the sender-identity check is too narrow | Deterministic rules | From-header match against authoritative sender list | ✅ Yes |
| `GENUINE_REPLY` | Business | Real prospect engagement requiring operator response | Human | Human composition + business intent | ⚠️ Partial (communication-layer detection helps narrow but final commit is human) |
| `WRONG_CONTACT` | Communication | Real human responded but they're not the decision-maker | Communication-layer classifier with high-confidence regex / AI | Patterns like "you should contact our facilities manager" + sender identity ≠ original contact | ✅ Yes |
| `INVALID_CONTACT` | Transport | Email address is structurally valid but the receiving server permanently rejects | Deterministic | DSN header (`Final-Recipient`, `Status: 5.x.x`) on inbound message | ✅ Yes |
| `ENTITY_CONFLICT` | Ingestion | Same email/contact assigned to multiple entities upstream | Deterministic at ingestion | UNIQUE constraint violation at entity-resolution layer | ✅ Yes (W2 territory) |
| `BOUNCED` | Transport | Hard bounce — recipient address doesn't exist | Deterministic | DSN with `5.1.1` or similar permanent-failure SMTP code | ✅ Yes |
| `CLASSIFIER_UNCERTAIN` | Communication | Message is human-authored but intent is ambiguous | Deterministic (this is the "default" bucket for low AI confidence) | AI returns valid intent but with low confidence score | ✅ Yes (as a routing decision, not as truth) |
| `PLACEHOLDER_CONTACT` | Ingestion | Email is syntactically valid but semantically fake (`name@domain.com`, `info@company.com`) | Deterministic at ingestion | Pattern denylist | ✅ Yes (W2 territory) |
| `INGESTION_CORRUPTION` | Ingestion | Transport/encoding artifacts leaked into business state (`%20info@`, `u003e...`, multi-address concatenation) | Deterministic at ingestion | Format regex + presence of encoded characters | ✅ Yes (W2 territory) |

Six of nine are deterministically auto-detectable from machine-readable signals. Only **GENUINE_REPLY** absolutely requires human commit, and even that has machine-assisted narrowing via communication-layer detection.

## §5 — Who should ever see what

Each terminal state maps to one of three operational dispositions: it should reach a human, it should be handled deterministically by infrastructure, or it requires operator cognition.

### Never reaches a human (deterministic infrastructure handles)
Categories where deterministic detection is reliable enough that no human review adds value:

| State | Why deterministic | Action |
|---|---|---|
| `SELF_FOLLOWUP` | Sender-identity match against authoritative AskMiro sender list is exact | Discard from queue; resume normal follow-up cadence |
| `INVALID_CONTACT` | DSN headers + RFC validation are deterministic | Auto-suppress; flag email as permanently bad |
| `BOUNCED` | DSN status codes are deterministic | Same as above |
| `PLACEHOLDER_CONTACT` | Pattern denylist is deterministic | Rejected at ingestion (W2); should never reach outreach |
| `INGESTION_CORRUPTION` | Format regex is deterministic | Rejected at ingestion (W2); should never reach outreach |

If these reach the human queue, **the system has a bug at the layer that should have caught them**. Their presence is itself a defect signal.

### Deterministic infrastructure (machine handles entirely)
| State | Authority | Audit trail required |
|---|---|---|
| `ENTITY_CONFLICT` | Entity-resolution layer (W2) at ingestion | Yes — log which conflicting source records produced the duplicate |
| `BOUNCED` | DSN handler in scanOutreachReplies | Yes — preserve DSN + suppression event |
| `SELF_FOLLOWUP` | Mail-direction classifier (W3) before AI invocation | Yes — log which sender identity matched |

### Requires operator cognition (humans handle)
Only these should ever appear in the human-review queue:

| State | Why humans | Expected volume per 100 leads contacted |
|---|---|---|
| `GENUINE_REPLY` | Business judgment — every reply is potential revenue. Even AI can't reliably distinguish "interested" from "polite decline" without context. | 5–15% (industry-realistic cold-outreach reply rate) |
| `WRONG_CONTACT` (when AI confidence is high but the redirect target isn't in our database) | Operator decides whether to research the new contact or close the lead | 2–8% |
| `CLASSIFIER_UNCERTAIN` (low-confidence AI output on a real human-authored message) | The AI explicitly admits it doesn't know — that's the case for a human read | <5% if classifier is well-tuned |

For a properly-functioning system, the human queue per day should be 7–25% of leads contacted that day. Not 100% of leads ever contacted. The current 60 (down from 305) is still inflated relative to that target because SELF_FOLLOWUP hasn't been deterministically excluded yet.

## §6 — Implications for W1 / W3

This document is the empirical input both workstreams have been waiting for.

### W1 (Queue taxonomy refactor)

The current schema is:
```
needsHumanAction: 'true' | 'false'
humanActionReason: free-text string
```

The replacement schema, derived empirically:
```
queueState: enum SELF_FOLLOWUP | GENUINE_REPLY | WRONG_CONTACT |
              INVALID_CONTACT | ENTITY_CONFLICT | BOUNCED |
              CLASSIFIER_UNCERTAIN | PLACEHOLDER_CONTACT |
              INGESTION_CORRUPTION
queueLayer: enum TRANSPORT | COMMUNICATION | BUSINESS | INGESTION
queueConfidence: enum HIGH | MEDIUM | LOW (only meaningful for COMMUNICATION/BUSINESS states)
queueEvidence: JSON — what evidence triggered this state
queueWriter: enum DETERMINISTIC | CLASSIFIER | OPERATOR
queueWriterVersion: string (rules version or classifier version)
queueWrittenAt: timestamp
```

The migration backfill is mostly deterministic because most current rows fall into the deterministic states. Run `_classifyNeedsAttention()` once, generate the queueState assignment per row, write to a new column without touching existing fields. Operators can see both the old `humanActionReason` and the new `queueState` during the transition.

### W3 (Classifier authority refactor)

The classifier should be restructured into three serial gates, each authoritative at its own layer:

1. **Pre-classifier (transport layer)** — runs first, deterministic:
   - Sender identity check (against authoritative AskMiro sender list)
   - DSN header check
   - Auto-submitted / Auto-reply header check
   - Outputs: `SELF_FOLLOWUP` | `BOUNCED` | `OUT_OF_OFFICE` | `PROCEED_TO_LAYER_2`

2. **Communication-layer classifier** — runs only if pre-classifier returns PROCEED:
   - AI interpretation with confidence stamp
   - Outputs: intent (`POSITIVE`/`NOT_INTERESTED`/`WRONG_CONTACT`/`INFO_REQUEST`/etc.) + confidence (`HIGH`/`MEDIUM`/`LOW`)

3. **Business-layer commit** — runs only on `HIGH` confidence + only for certain intents:
   - `POSITIVE` / `INFO_REQUEST` → human-only commit (queue as GENUINE_REPLY)
   - `NOT_INTERESTED` / `UNSUBSCRIBE` → deterministic commit (auto-close with audit)
   - `WRONG_CONTACT` → deterministic commit if redirect target is identifiable, else human
   - Any `LOW` / `MEDIUM` → queue as CLASSIFIER_UNCERTAIN

Each layer writes to its own audit trail (the `outreach_events` table from migration 010, currently unmigrated but designed). Each decision records: timestamp, layer, classifier version, confidence, evidence snippet, resulting state.

The current `_ruleBasedClassify` and `_classifyReplyAI` functions can be reused — they become the implementations of step 2. What's missing is step 1 (pre-classifier) and step 3 (confidence-gated business commit).

The dormant `MAIL_DIRECTION_CLASSIFIER_ENABLED` flag is the kill switch for step 1. The dormant `BOUNCE_HANDLER_ENABLED` flag is the kill switch for the deterministic commits in step 3.
